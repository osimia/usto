# Чеклист производительности: Flutter + Go
### Как профи делает «шустрое» приложение — от рендера до базы данных
> Контекст: маркетплейс услуг (лента заявок с фото, чат, кошелёк), рынок ЦА = бюджетные Android + медленный/дорогой интернет. Поэтому приоритеты: размер APK, картинки, устойчивость к 3G, экономия батареи.

---

## 0. Бюджеты производительности (сначала цели, потом оптимизации)

- [ ] Холодный старт до первого осмысленного кадра: **< 2 сек** на Redmi/Samsung A-серии (не на твоём флагмане!)
- [ ] Рендер: **60 fps**, jank (кадры > 16,6 мс) **< 1%** на профиле реального бюджетника
- [ ] API: **p95 < 200 мс, p99 < 500 мс** на критичных эндпоинтах (лента, создание заявки, отклик)
- [ ] Размер APK (arm64): **< 25 МБ**; каждый релиз — контроль дельты в CI
- [ ] Экран ленты работает на **3G** (throttling в тестах) и переживает обрыв сети без крэша
- [ ] Crash-free rate: **> 99,5%**
- [ ] Расход батареи: приложение не висит в фоне с открытым сокетом без нужды

Правило: любая оптимизация начинается с измерения (DevTools / pprof / Grafana), а не с догадки.

---

## 1. Flutter: rebuild'ы и архитектура виджетов

- [ ] `const` конструкторы везде, где возможно (включи линтер `prefer_const_constructors` как error)
- [ ] Гранулярный state: Riverpod/Bloc с точечной подпиской (`ref.watch(provider.select(...))`), а не `setState` на весь экран
- [ ] Никаких «функций-виджетов» (`Widget _buildCard()`) для крупных кусков — только отдельные классы-виджеты: у них работает кэширование element tree
- [ ] `RepaintBoundary` вокруг элементов, которые перерисовываются независимо (карточка в ленте, бейдж таймера, анимированный счётчик)
- [ ] Дорогие вычисления не в `build()` — build должен быть чистым и дешёвым, вызываться может сотни раз
- [ ] `AnimatedBuilder`/`ListenableBuilder` с параметром `child:` — анимируется обёртка, тяжёлое поддерево не пересобирается
- [ ] Ключи (`ValueKey`) на элементах списков с изменяемым порядком (отклики при пересортировке)
- [ ] DevTools → Performance → «Track widget rebuilds»: на экране ленты при скролле не должно пересобираться ничего, кроме видимых карточек

## 2. Flutter: списки (лента заявок — сердце приложения)

- [ ] Только `ListView.builder` / `SliverList` — никогда `ListView(children: [...])` и никогда `shrinkWrap: true` в длинных списках
- [ ] `itemExtent` или `prototypeItem`, если карточки одинаковой высоты — скролл-математика становится O(1)
- [ ] `CustomScrollView` + slivers для сложных экранов (шапка + фильтры + лента)
- [ ] `cacheExtent` настроен разумно (1–2 экрана вперёд), prefetch следующей страницы при достижении ~70% скролла
- [ ] Бесконечная лента: пагинация подгружается фоном, скелетоны вместо спиннера на весь экран
- [ ] Избегай `Opacity`, `ClipRRect`, `ShaderMask` внутри item'ов списка (вызывают saveLayer): полупрозрачность — через цвет с альфой, скругления — через `DecoratedBox`/`Card`

## 3. Flutter: изображения (главный пожиратель памяти и трафика)

- [ ] `cached_network_image` (или свой слой на `flutter_cache_manager`): диск-кэш обязателен
- [ ] **Сервер отдаёт нужный размер**: превью в ленте ≈ 300–400px WebP (~15–30 КБ), полное фото — по клику. Никогда не грузить оригинал 4 МБ в карточку 100×100
- [ ] `cacheWidth`/`cacheHeight` при декоде = размеру виджета × devicePixelRatio — иначе 4000px фото декодится в память целиком
- [ ] Плейсхолдеры: blurhash/thumbhash (сервер считает при загрузке фото) — лента выглядит мгновенной
- [ ] Лимит memory-кэша картинок (`PaintingBinding.instance.imageCache.maximumSizeBytes` ≈ 50–100 МБ на бюджетниках)
- [ ] Загрузка фото заявки пользователем: сжатие НА КЛИЕНТЕ до отправки (`flutter_image_compress`, ~1280px, quality 80) — экономит трафик юзера и твоё хранилище

## 4. Flutter: сеть и данные

- [ ] `dio` с keep-alive; включи gzip/brotli (заголовок `Accept-Encoding`) — JSON жмётся в 5–10 раз
- [ ] Курсорная пагинация везде (лента, отклики, чат, транзакции) — см. п. 8 про backend
- [ ] Debounce поиска 300–400 мс + отмена предыдущего запроса (`CancelToken`)
- [ ] **Optimistic UI**: отклик/сообщение/лайк рисуются мгновенно, откат при ошибке — ощущение скорости важнее скорости
- [ ] Offline-first кэш справочников и последней ленты: Isar/Drift/Hive; паттерн stale-while-revalidate (показать кэш → тихо обновить)
- [ ] Retry с экспоненциальным backoff + jitter на GET; на POST — только с идемпотентным ключом (см. п. 10)
- [ ] Парсинг больших JSON (>100 КБ) в изоляте: `compute(parseOrders, body)` — иначе фриз кадров
- [ ] Один источник правды о состоянии сети (`connectivity_plus`) + баннер «офлайн», очередь исходящих действий

## 5. Flutter: старт и размер приложения

- [ ] `main()` максимально тонкий: тяжёлая инициализация (аналитика, удалённый конфиг) — лениво/параллельно ПОСЛЕ первого кадра
- [ ] Нативный splash (`flutter_native_splash`), без белой вспышки
- [ ] Релиз только как App Bundle / `--split-per-abi`; проверка: `flutter build apk --analyze-size`
- [ ] `--tree-shake-icons`, `--obfuscate --split-debug-info` (плюс безопасность)
- [ ] Шрифты: только нужные веса, subset кириллица+латиница; без пакетов «всё включено»
- [ ] Deferred components для редких экранов (например, весь флоу верификации мастера)
- [ ] Аудит зависимостей раз в квартал: каждая либа = мегабайты и время старта (особенно берегись firebase_* набора «на всякий случай»)

## 6. Flutter: рендер, jank и профилирование

- [ ] Профилируем ТОЛЬКО в `--profile` на реальном бюджетном Android (эмулятор и debug-режим врут)
- [ ] Performance overlay: обе полосы (UI и Raster) стабильно зелёные при скролле ленты и открытии заявки
- [ ] Impeller включён (дефолт) — но прогони на 2–3 старых девайсах, при артефактах есть флаг отката
- [ ] Hero-анимации и переходы: 60fps; тяжёлый экран строится ПОСЛЕ завершения перехода (`Future.delayed` + placeholder или `TickerMode`)
- [ ] Таймер/стрим в карточке («5 мин назад») не должен перерисовывать список — один общий тикер + локальные RepaintBoundary
- [ ] Golden-правило чата: виртуализация сообщений, картинки лениво, скролл к последнему без построения всей истории

## 7. Чат и realtime (не убить батарею и сервер)

- [ ] WebSocket с reconnect + экспоненциальный backoff + jitter; heartbeat ping/pong 30–60 с
- [ ] В фоне сокет закрывается, доставка через FCM data-messages; при открытии — дельта-синхронизация (`since=last_message_id`)
- [ ] Локальная очередь исходящих сообщений (статусы: отправляется → доставлено → ошибка/повтор)
- [ ] MVP-альтернатива: обычный polling каждые 10–15 с на открытом чате — проще и часто достаточно; сокеты — когда появится нагрузка

## 8. Go: HTTP-слой

- [ ] `http.Server` с явными таймаутами: `ReadHeaderTimeout: 5s`, `ReadTimeout: 10s`, `WriteTimeout: 15s`, `IdleTimeout: 60s` (дефолты = ноль = дыра под медленных клиентов)
- [ ] Middleware-цепочка: recovery (panic → 500 + лог), request ID, gzip/brotli, access-log, rate limit
- [ ] Ответы — узкие DTO под экран, а не «вся модель из базы»: лента отдаёт 10 полей карточки, не 40 полей заявки
- [ ] Graceful shutdown (`signal.NotifyContext` + `srv.Shutdown`) — деплой без обрыва запросов
- [ ] `net/http/pprof` на отдельном внутреннем порту — профилирование CPU/heap в проде за минуту
- [ ] `context.Context` с таймаутом сквозь весь стек: хендлер → сервис → БД → внешний API; ни одного вызова без дедлайна
- [ ] JSON: стандартного `encoding/json` хватает до серьёзных нагрузок; не тащи sonic/easyjson преждевременно

## 9. Go: PostgreSQL (где живёт 80% латентности)

- [ ] `pgx/v5` + `pgxpool`; размер пула: `min(4 × CPU, ~20–30)`, метрики пула (busy/idle/wait) в Grafana — исчерпание пула это «почему всё зависло»
- [ ] Индексы под реальные запросы ленты:
  ```sql
  -- лента мастера: мои категории + мой город, свежие сверху, только активные
  CREATE INDEX idx_orders_feed ON orders (city_id, category_id, created_at DESC)
    WHERE status = 'active';
  -- отклики заявки
  CREATE INDEX idx_responses_order ON responses (order_id, created_at);
  ```
- [ ] **Keyset-пагинация вместо OFFSET** (OFFSET 5000 читает и выбрасывает 5000 строк):
  ```sql
  WHERE (created_at, id) < ($cursor_ts, $cursor_id)
  ORDER BY created_at DESC, id DESC LIMIT 20;
  ```
- [ ] N+1 запрещён: карточки ленты собираются одним запросом с JOIN'ами/подзапросами (счётчики, имя мастера), не циклом
- [ ] Счётчики (`views_count`, `responses_count`) денормализованы: `UPDATE ... SET responses_count = responses_count + 1` в той же транзакции, что и вставка отклика
- [ ] `EXPLAIN (ANALYZE, BUFFERS)` на топ-10 запросов до релиза; `pg_stat_statements` включён в проде с первого дня
- [ ] Кошелёк: списание за отклик — одна короткая транзакция с `SELECT ... FOR UPDATE` строки баланса; баланс никогда не уходит в минус на уровне БД (`CHECK (balance >= 0)`)
- [ ] Никаких долгих транзакций, охватывающих внешние вызовы (SMS/пуш внутри транзакции = блокировки на секунды)
- [ ] Миграции — golang-migrate/goose, каждая обратима

## 10. Go: конкурентность, очереди, надёжность

- [ ] Хендлер НИКОГДА не делает синхронно: отправку SMS, пуши, ресайз фото, письма. Всё — в очередь (asynq/River на Redis/PG), воркеры отдельно
- [ ] Схема отклика: транзакция (списание + вставка отклика + счётчик) → commit → событие в очередь → воркер шлёт пуш заказчику. API отвечает за <100 мс
- [ ] `errgroup` + семафоры для параллельных внешних вызовов; на каждый внешний API — свой таймаут (SMS-шлюз: 3 с, не 30)
- [ ] **Идемпотентность денег**: заголовок `Idempotency-Key` на пополнении и отклике; уникальный индекс по ключу; повторный запрос возвращает первый результат, а не второе списание
- [ ] Rate limiting: per-user на отклики/создание заявок (антиспам), per-IP на auth/SMS (антифрод и защита SMS-бюджета)
- [ ] Вебхуки платёжного шлюза: проверка подписи, идемпотентная обработка, ответ 200 быстро, обработка в воркере
- [ ] `-race` в CI; нагрузочный прогон k6/vegeta критичных эндпоинтов ДО запуска (цель: лента держит 200 RPS на одном инстансе)

## 11. Go: кэширование

- [ ] Redis: горячая лента первых страниц (TTL 10–30 с — этого достаточно, не усложняй инвалидацию), справочник категорий/районов (TTL 1 ч + сброс по событию), rate-limit счётчики, сессии
- [ ] In-process кэш (ristretto / sync.Map) для категорий — избавляет даже от похода в Redis
- [ ] `ETag`/`If-None-Match` на справочниках — клиент получает 304 и ноль байт
- [ ] Кэшируй результат, а не промисы решений: не кэшировать баланс кошелька и статусы откликов (деньги и статусы — всегда из БД)

## 12. Картинки на бэке

- [ ] Upload → S3/MinIO через presigned URL (файл не проходит через Go-сервер), лимит 10 МБ, проверка magic bytes
- [ ] Воркер делает 3 размера (thumb 320 / medium 800 / full 1600) + WebP + blurhash; в БД — ключи всех размеров
- [ ] Отдача только через CDN (Cloudflare бесплатного тарифа хватит надолго), кэш навсегда: имя файла = content hash
- [ ] EXIF-очистка (гео-метки из фото заявок = приватность заказчика!)

## 13. Наблюдаемость (без неё «шустрость» не удержать)

- [ ] Prometheus + Grafana: RED-метрики каждого эндпоинта (rate / errors / duration p50-p95-p99), метрики пула БД, длина очередей воркеров, лаг пушей
- [ ] Алерты: p95 > 300 мс 5 минут, errors > 1%, пул БД в wait, очередь растёт
- [ ] Sentry в Go И во Flutter (crash + ANR); sourcemaps/symbols загружаются в CI
- [ ] Firebase Performance или собственные метрики во Flutter: холодный старт, TTI ленты, время фото-аплоада — с реальных устройств пользователей
- [ ] Structured logging (`slog`/zerolog) + сквозной trace_id: мобилка шлёт `X-Request-ID`, он же в логах Go и в ответе — инцидент разбирается за минуты
- [ ] Дашборд «золотой путь»: регистрация → заявка → отклик → чат, конверсия и латентность каждого шага

## 14. Инфраструктура

- [ ] Go = один статический бинарь в минимальном Docker-образе (distroless/alpine, ~15 МБ)
- [ ] Reverse proxy Caddy/Nginx: TLS 1.3, HTTP/2, brotli, лимиты тела запроса
- [ ] Регион хостинга: Франкфурт/Алматы — замерь RTT из Душанбе до кандидатов (обычно 60–120 мс), это база всей латентности
- [ ] PgBouncer перед Postgres при росте (transaction pooling)
- [ ] Бэкапы PG: ежедневный полный + WAL-архив, **проверенное восстановление** (бэкап без теста восстановления = лотерея)
- [ ] Staging = уменьшенная копия прода; фиче-флаги для рискованных изменений

## 15. Процесс: как не потерять скорость через полгода

- [ ] CI-гейт: размер APK (падение сборки при +2 МБ без объяснения), `flutter analyze`, `go vet`, `-race`
- [ ] Ежемесячный «перф-день»: профилирование на бюджетном девайсе + 3G throttling + разбор топ-10 медленных запросов из `pg_stat_statements`
- [ ] Перф-бюджеты из раздела 0 висят в README и проверяются перед каждым релизом
- [ ] Каждая новая фича в ленте/чате проходит вопрос: «сколько это стоит в кадрах, байтах и запросах?»

## ТОП-10 «если делать только это» (по соотношению эффект/усилия)

1. Серверный ресайз фото + WebP + CDN (п. 3, 12) — минус 80% трафика и памяти
2. `ListView.builder` + const + гранулярный state (п. 1–2) — плавная лента
3. Keyset-пагинация + правильные индексы (п. 9) — API не деградирует с ростом данных
4. Всё тяжёлое — в воркеры через очередь (п. 10) — API всегда < 100–200 мс
5. Optimistic UI + скелетоны (п. 4) — ощущение мгновенности даже на 3G
6. Таймауты и context везде (п. 8) — один тормозящий SMS-шлюз не кладёт всё приложение
7. Сжатие фото на клиенте до аплоада (п. 3) — юзер на 3G не ждёт минуту
8. Идемпотентность денежных операций (п. 10) — ни одного двойного списания
9. pprof + pg_stat_statements + Sentry с первого дня (п. 13) — проблемы видны до жалоб
10. Профилирование на реальном бюджетном Android (п. 6) — твои пользователи живут там, а не на твоём девайсе
