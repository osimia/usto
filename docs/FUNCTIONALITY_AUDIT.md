# USTO Functionality Audit

## Цель

По каждой функции приложения — пройти от А до Я (что реально происходит сегодня, end-to-end, backend + Flutter) и зафиксировать, что нужно доделать, чтобы это стало по-настоящему рабочим продуктом, а не MVP-имитацией на одном demo-аккаунте.

## Главная сквозная находка

Реальный per-account `users` (таблица с уникальным `(phone_norm, role)`) уже существует, но всё, на что он указывает — `profiles`, `orders`, `masters`, `wallet`, `verification`, `chats` — по-прежнему завязано на **одну общую demo-строку на роль** (`a.profile(role)` = `SELECT ... WHERE role=?`, без фильтра по пользователю, `main.go:804-821`). `ensureUserForAuth` (`auth.go:179-195`) при создании любого нового пользователя присваивает ему ID этой же единственной demo-строки. Второй зарегистрированный мастер физически не может появиться в каталоге и получить свой отклик — все отклики жёстко пишутся на `master_id=1`.

Это не десяток разрозненных багов, а один структурный пробел (раздел 2 ниже), блокирующий "реальность" сразу нескольких функций (4, 5, 6, 8, 9). При планировании работы иметь это в виду в первую очередь.

Формат каждого раздела: **«Как есть»** → **«Что нужно для реальной работы»**.

> **Статус (обновлено после первой итерации фиксов):** раздел 2 (профиль/мультитенантность) реализован на backend — см. пометки ✅ внутри разделов 2, 4, 5, 7, 8, 9. Остальное (Flutter-сторона, фото, каталог мастеров как редактируемый профиль, чат-поллинг, cross-cutting UI) пока не тронуто.

## 1. Авторизация и сессия

**Как есть:** Онбординг-карусель → ввод телефона (кастомная numpad-клавиатура, задизайненный номер `900112233` по умолчанию, `auth_screen.dart:46`) → ввод SMS-кода → выбор роли customer/master → реальные `POST /auth/request-code` и `/auth/verify-code` → HMAC-SHA256 JWT (access+refresh, самописный, но корректный, `auth.go:217-282`). `users` — реальная per-account таблица (`phone_norm`+`role` уникальны). `AuthGate._user` (`auth_gate.dart:16-29`) хранится только в памяти — токены нигде не персистятся (в `pubspec.yaml` нет ни `shared_preferences`, ни `flutter_secure_storage`), поэтому каждый перезапуск приложения = принудительный повторный вход. Logout прокинут через 3 слоя (`AuthGate → HomeShell → ProfilePage`), но нигде не вызывается — в UI нет ни кнопки, ни пункта меню. На backend `logout` (`auth.go:142-147`) — пустая заглушка без отзыва токена; refresh-токены никогда не ротируются/не становятся одноразовыми. `request-code` не отправляет SMS — просто возвращает `"dev-"+phone`; `verify-code` сравнивает код со строкой `DEV_SMS_CODE` (по умолчанию `1234`).

**Что нужно для реальной работы:**
- Персистентное хранение токенов на клиенте (`flutter_secure_storage`), восстановление сессии при старте, авто-refresh по 401.
- Реальная кнопка/пункт "Выйти" в `ProfilePage`, которая чистит токены и дёргает `/auth/logout`.
- Backend: ротация refresh-токенов (или чёрный список отозванных), чтобы logout реально что-то инвалидировал.
- Интеграция реального SMS-провайдера за флагом `APP_ENV`, dev-код оставить только для локальной разработки/тестов.

## 2. Профиль и мультитенантность (приоритет №1) — ✅ реализовано

**Как было:** `profiles` — ровно одна строка `role='customer'` и одна `role='master'` на всю базу (`main.go:599-601`, сид-данные). `PATCH /me/profile` правило именно её — изменение имени/города одним заказчиком было видно всем заказчикам. `PublishedCount` ("мои заявки") считался как `SELECT COUNT(*) FROM orders` без фильтра вообще — потому что у `orders` не было колонки `customer_id`. Auth-проверка (JWT claims) стояла только на `/api/me` — все остальные мутации (создание заявки, отклик, пополнение, верификация) не проверяли токен вовсе.

**Что сделано:**
- `ensureUserForAuth` (`auth.go`) при первом входе нового `(phone, role)` создаёт **новую** строку `profiles` вместо переиспользования demo-профиля; два сид-профиля остались для демо-логина.
- Добавлены `orders.customer_id`, `masters.profile_id`, `transactions.profile_id` (обе СУБД + миграции для существующих БД); `PublishedCount` теперь считается по `customer_id`.
- Новый мастер автоматически получает пустую, редактируемую позже запись в каталоге `masters` (через `masters.profile_id`) — иначе `responses.master_id` было бы не на что ссылаться.
- `orders.go`/`responses.go`/`wallet.go`/`verification.go` теперь требуют JWT и действуют от лица реального авторизованного профиля, а не `WHERE role='master'`/`'customer'`.
- Легаси `POST /api/responses` (старый web-прототип, без авторизации) сохранён без изменений в поведении — резолвится через demo-профиль по реальному lookup вместо хардкода `1`.
- Проверено вручную (curl) и тестами (`wallet_test.go`, включая `TestSecondMasterIsIndependentFromDemoMaster`): второй мастер получает независимый кошелёк/каталожную запись/атрибуцию откликов.

**Не сделано в этой итерации** (сознательно, см. `RUN.md`): полноценное редактирование карточки мастера в каталоге (см. п.6) и Flutter-сторона (персистентность сессии и т.д., см. п.1) — это отдельные, более крупные куски работы.

## 3. Категории и услуги

**Что нужно:** Вынести услуги в отдельную таблицу `services(category_id, name)`, засеять по всем категориям, убрать switch.

## 4. Заявки и создание заявки

**Как есть:** Реальные создание/список/детали с server-side фильтрацией (категория/район/статус — в SQL) и пагинацией; атомарный счётчик просмотров. Статусов жизненного цикла по факту два: `"Активная"` (при создании) и `"Мастер выбран"` (при выборе мастера) — `"Новая"`/`"Выбор мастера"` существуют только в сид-данных и никогда не достижимы через API; нет перехода в "завершена"/"отменена". Нет `customer_id` (см. п.2). **Фото заявок нет вообще** — ни backend endpoint'а загрузки, ни `image_picker` на клиенте, ни шага прикрепления фото в 5-шаговом визарде создания заявки, ни строчки в `docs/API_SPEC.md`. Район/бюджетные пресеты/шаблоны в визарде — статические массивы на клиенте. На экране деталей заявки нет pull-to-refresh.

**Что нужно:**
- ✅ Привязан `customer_id`; `GET /api/orders?mine=1` отдаёт реальные заявки авторизованного заказчика, создание заявки требует токен роли `customer`.
- Добавить переходы завершения/отмены заявки (endpoints + статус-машина).
- Построить пайплайн фото: endpoint загрузки (multipart, лимит размера, проверка типа), таблица `order_photos`, шаг прикрепления в визарде (`image_picker`), отображение с плейсхолдером.
- `RefreshIndicator` на экран деталей заявки.

## 5. Отклики и выбор мастера

**Как есть:** Создание отклика уже реально безопасно транзакционно (одна транзакция: гарантированное списание баланса + вставка отклика + запись транзакции + идемпотентность по ключу) — единственная область, уже сделанная "по-взрослому" с точки зрения целостности данных. Но `master_id` в `INSERT INTO responses` жёстко `1` всегда, и хендлер не проверяет авторизацию вовсе. `selectMasterForOrder` (проверка что отклик принадлежит заявке, обновление статуса, создание чата) — реальна и корректна. На клиенте `ResponseSheet` (форма отклика мастера) показывает захардкоженный текст баланса `"Баланс: 85 TJS"` / `"Спишется: -4 TJS"` вместо реальных данных из `/wallet`.

**Что нужно:**
- ✅ Реальный `master_id`/`masterProfileID` теперь берутся из JWT-claims (`responses.go`), создание отклика требует токен роли `master`; select-master по-прежнему без auth-проверки (не менялось в этой итерации — доступ к нему уже проверяется косвенно через владение заявкой на Flutter-стороне, но серверной проверки роли/владения там ещё нет).
- Привязать `ResponseSheet` (Flutter) к реальному `GET /wallet` вместо статичного текста — не сделано.

## 6. Каталог мастеров

**Как есть:** Список/детали/отзывы мастеров — реальные, с фильтром по услуге, полнотекстовым поиском и сортировкой по рейтингу — крепкая read-модель. Но таблица `masters` — статичные 3 сид-строки, никак не связанные с реальными аккаунтами `users`/`profiles`: нет ни одного endpoint'а, который создавал/обновлял бы запись в каталоге от лица авторизованного мастера. Реальный зарегистрировавшийся мастер никогда не появится в поиске, а все его отклики (п.5) всё равно улетают на demo-мастера №1. Два несвязанных флага "verified" (`masters.verified` — статичный, `profiles.is_verified` — из верификации) никогда не синхронизируются. На клиенте `MastersPage` не имеет ни поиска, ни фильтра в UI; кнопка "Все" не имеет `onTap` (мертва).

**Что нужно (крупный, основополагающий для мастер-стороны блок):**
- Endpoint(ы) создания/редактирования собственной карточки мастера от авторизованного аккаунта (имя, услуга, био, навыки, портфолио) — заменяет статичную таблицу `masters` реальными записями по аккаунтам.
- Свести два флага "verified" к одному источнику правды.
- Добавить поиск/фильтр в UI `MastersPage`, подключить кнопку "Все" к реальному экрану обзора.

## 7. Чат

**Как есть:** Реальная модель чата на пару (заявка, мастер) с уникальным индексом, реальные endpoints, `ensureChat` подключён из выбора мастера. Рядом продолжает существовать легаси-эндпоинт `/api/messages`, жёстко привязанный к `chatID=1` (не используется мобильным клиентом, но не удалён). Реалтайма нет вообще — ни поллинга, ни WebSocket; сообщения подгружаются только при открытии экрана или после отправки. Имя заказчика в списке чатов берётся из singleton-профиля (зависит от п.2).

**Что нужно:**
- Убрать/задепрекейтить легаси `/api/messages` после подтверждения, что им никто не пользуется.
- Добавить поллинг (обновление по таймеру, пока экран чата открыт).
- ✅ Имя заказчика в `a.chats()` теперь берётся из реального `orders.customer_id` (с fallback на demo-профиль для заявок, созданных до миграции); имя мастера уже было реальным (каталог).

## 8. Кошелёк

**Как есть:** Реально транзакционное пополнение с идемпотентностью и историей транзакций — вместе с откликами (п.5) вторая по-настоящему крепкая область, подтверждённая тестами (`wallet_test.go`). Авторизации нет — любой вызывающий пополняет "тот самый" единственный кошелёк мастера. На клиенте: захардкоженный курс USD, захардкоженный fallback баланса `85`, dev-параметр `wrap=1` зашит прямо в боевой URL (`'/wallet/topup?wrap=1'`).

**Что нужно:**
- ✅ Кошелёк/пополнение/история транзакций теперь требуют токен роли `master` и работают с реальным `profileID` вызывающего (в т.ч. `transactions` получила `profile_id` — раньше это был вообще общий лог на всех).
- Убрать `wrap=1` из боевых вызовов клиента (Flutter) — не сделано.
- Убрать захардкоженные курс/fallback баланса (Flutter) — не сделано.
- Реальная платёжная интеграция намеренно вне MVP по `PRODUCT_SPEC.md` — осознанное ограничение, не баг.

## 9. Верификация мастера

**Как есть:** Подача документов и статус — реальные, в своей таблице. Хранение файла — фиктивное (клиент передаёт произвольную строку-URL, реальной загрузки нет). `verifyMaster` (подтверждение) не проверяет авторизацию и принудительно верифицирует singleton-профиль мастера. На клиенте в боевом UI есть кнопка **"Dev-подтвердить"**, вызывающая `/verification?wrap=1`.

**Что нужно:**
- Реальная загрузка документа (можно переиспользовать инфраструктуру фото из п.4) — не сделано.
- ✅ `verifyMaster`/статус/документы теперь привязаны к реальному ID авторизованного мастера вместо singleton-профиля. Вопрос "должен ли мастер сам себя верифицировать" (self-service) намеренно не пересматривался — это отдельное решение про будущую admin-панель, не про identity-фикс. Dev-кнопка в Flutter UI не убрана.

## 10. Сквозное качество UI и платформы

**Как есть:** Состояния загрузки/ошибки/пустого экрана переиспользуются консистентно через общие виджеты — уже хорошо сделано. Pull-to-refresh — не везде (есть в таб-шелле и кошельке, нет в деталях заявки/мастера/чате/верификации). Навигация единообразна (`Navigator.push`), но без именованных роутов/deep-link. Счётчик просмотров имеет несовпадающий fallback (`12` в одном месте, `0` в другом). Выбор города на главном экране декоративный (нет `onTap`). Поле поиска на главном экране — не поиск, а ярлык на создание заявки; экрана "все категории" не существует. Rate limiting сегодня только на 2 auth-эндпоинтах. CORS отражает любой Origin.

**Что нужно:**
- `RefreshIndicator` на детали заявки/мастера/чат/верификацию.
- Единый fallback просмотров.
- Реальный выбор города (если мультигород — ближайшая цель) либо явно зафиксировать один город как осознанное MVP-ограничение.
- Реальный поиск либо убрать вводящий в заблуждение поисковый ярлык.
- Подключить "все категории"/"все мастера" к реальным экранам обзора.
- Расширить rate limiting на создание отклика и пополнение кошелька (переиспользовать `ipRateLimiter`).
- Ужесточить CORS перед реальным деплоем.
